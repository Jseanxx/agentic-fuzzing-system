# Fuzzing Records Index

이 폴더는 `fuzzing-jpeg2000`에서 만들고 있는 **자가발전형 퍼징 시스템**의 기록 저장소다.
목적은 단순 로그 적재가 아니라:

- 어떤 단계까지 왔는지 추적
- 각 단계가 실제로 가치 있었는지 나중에 재평가
- 실패/병목/가정 오류를 다시 학습 재료로 사용
- 시스템이 잘못된 방향으로 가도 되돌릴 수 있게 **버전처럼 관리**
- Hermes가 다음 개선안을 더 잘 제안할 수 있게 근거를 축적

즉, 이 폴더는 실행 로그 보관함이 아니라
## **“에이전트 퍼징 시스템의 버전 관리 + 피드백 메모리”**
역할을 한다.

---

## 먼저 읽을 문서

### 1. 지금 상태를 한 장으로 보기
- `current-status.md`
  - 현재 단계, 최근 완료 단계, 남은 핵심 단계, 냉정한 평가

- `.hermes/plans/2026-04-15_133343-semi-autonomous-multi-target-fuzzing-roadmap.md`

## 폴더 구조

### `current-status.md`
최신 한 줄 상태 문서.
- 지금 어디까지 왔는지
- 최근 완료 단계
- 남은 핵심 단계
- 점수화/냉정 평가

### `progress-index.md`
상위 진행표.
- 단계별 버전형 이력
- 각 단계의 의미
- 관련 note/checklist 링크
- 다음 단계 우선순위

### `notes/`
단계별 결과와 판단 메모.
권장 용도:
- 왜 이 단계가 필요했는지
- 무엇이 붙었는지
- 한계/리스크
- 다음 단계 힌트

### `checklists/`
실행 체크리스트.
권장 용도:
- 구현/테스트/검증 단계 관리
- 성공 기준 / 실패 기준 기록
- 작업 완료 여부를 사람이 재검토 가능하게 남기기

### `plans/`
작업 계획 문서.
권장 용도:
- 큰 기능 추가 전 실행 순서 문서화
- 구현 전에 단계 분해
- 하위 작업이나 원격 실행 계획 정리

### `runs/`
개별 런 기록.
권장 용도:
- 특정 실행의 결과 보존
- coverage/crash/no-progress/build-fail 등 런 단위 확인

### `policies/`
운영 정책 문서.
권장 용도:
- 시스템이 어떤 기준으로 행동해야 하는지 명시
- 위험한 자동화 억제
- retry/escalation 기준 정리

### `limitations/`
현재 한계 기록.
권장 용도:
- 왜 아직 완전 자율형이 아닌지
- 어떤 병목이 있는지
- 다음 구조 개선의 근거

### `prompts/`
상위 목적/외부 공유용 프롬프트 및 설명 자료.

## 기록 원칙

### 1. 최신 상태는 짧고 강하게
- `current-status.md`는 항상 최신 관점 유지
- 오래된 내용 누적보다 현재 판단이 우선

### 2. 개별 단계는 note + checklist 쌍으로 남기기
- `notes/`: 의미와 해석
- `checklists/`: 실행 검증

### 3. “붙였다”보다 “왜 가치 있었는지”를 남기기
좋은 기록 예:
- 어떤 루프가 새로 닫혔는가
- 무엇이 아직 heuristic 수준인가
- 어떤 리스크가 줄었는가

### 4. 실패도 버전처럼 남기기
이 시스템은 나중에 잘 될 보장이 없으므로,
성공한 단계만이 아니라:
- 틀린 가정
- 과잉 구조화
- 실제 finding efficiency와 무관했던 단계
도 다음 판단 재료로 남긴다.

### 5. 되돌릴 수 있게 링크 중심으로 남기기
각 상위 문서는 관련 note/checklist를 링크해,
나중에 “왜 이런 결론이었지?”를 추적 가능해야 한다.

---

## 현재 기록 상태에 대한 메모
현재는 문서 **축적은 많이 되어 있지만**, 최신 상태를 한눈에 보여주는 입구 문서는 약했다.
그래서 이번 정리에서:
- `README.md`를 인덱스 문서로 강화
- `current-status.md`를 최신화
- `progress-index.md`를 추가
하는 방향으로 재구성했다.
